# 📚 书签工具

## 创建书签

**触发方式**（任意一种）：
- "创建书签 `<名称>`" / "添加书签 `<名称>`" / "新建书签 `<名称>`"
- "记个书签 `<名称>`" / "添加个书签 `<名称>`" / "做个书签 `<名称>`"
- "创建书签" / "添加书签" / "新建书签"（不指定名称时询问）

**执行步骤**：

### 步骤 1：自动获取上下文
- 从 `PROGRESS.md` 获取当前"进行中"的模块
- 从模块的 `checklist.md` 获取当前学习位置（最后一个有勾选的章节）

### 步骤 2：询问用户
- 疑问点描述（想要探索什么问题？）
- 探索目标（可选，系统可智能生成）

### 步骤 3：智能生成内容
根据疑问点类型，自动生成探索目标和返回验证：

| 疑问类型 | 探索目标 | 返回验证 |
|----------|----------|----------|
| 命令相关 | 阅读文档、实践使用、记录技巧 | 熟练使用该命令 |
| 概念相关 | 阅读资料、整理笔记 | 能够解释概念 |
| 配置相关 | 查看文档、实践配置 | 能够独立配置 |
| 机制相关 | 阅读源码、实验验证 | 能够说明工作原理 |

### 步骤 4：显示预览
```
╔═══════════════════════════════════════════════════════╗
║                    书签预览                           ║
╠═══════════════════════════════════════════════════════╣
║ 名称：深入理解 Tool Use 机制                          ║
║ 来源模块：ai-tools-fundamentals                       ║
║ 主线位置：完整模式 > 第一部分 > 架构与概念            ║
╠═══════════════════════════════════════════════════════╣
║ 疑问点：                                              ║
║   Tool Use 的具体实现原理是什么？                      ║
╠═══════════════════════════════════════════════════════╣
║ 探索目标：                                            ║
║   - [ ] 阅读官方 Tool Use 文档                        ║
║   - [ ] 查看 Tool Use 的源码实现                      ║
╠═══════════════════════════════════════════════════════╣
║ 返回验证：                                            ║
║   - [ ] 能够解释 Tool Use 的工作原理                  ║
╚═══════════════════════════════════════════════════════╝
```

### 步骤 5：确认并保存
用户确认后，将书签添加到 `LEARNING_BOOKMARKS.md`

**触发方式**（任意一种）：

   ```

   ├── README.md           # 缓存说明

   ├── concepts.md         # 核心概念
   ```

**优化参数说明**：
- `retain_images=false`：跳过图片下载，提升加载速度 50-70%

# 📢 缓存更新提示系统（v4.0 新增）

> 当刷新知识缓存时，自动检测内容变更并生成课程修改建议

## 工作原理

1. **刷新缓存**时读取新旧版本的 `.metadata.json`
2. **对比 update_notes** 识别版本间的变更
3. **生成变更建议列表**并展示给用户
4. **用户确认后**执行变更并记录到 `course-updates.md`

## 执行流程

### 步骤 1：刷新缓存

执行正常的缓存刷新流程（见上方"刷新知识缓存"）

### 步骤 2：检测变更

对比新旧版本的 `.metadata.json`：

```
对比维度：
• topics_covered - 新增/删除的主题
• version - 版本号变化
• update_notes - 更新说明
```

### 步骤 3：生成变更建议

```
╔═══════════════════════════════════════════════════════╗
║  📢 缓存更新检测到以下变更                             ║
╠═══════════════════════════════════════════════════════╣
║ 版本：v1.1 → v1.2                                      ║
║ 更新时间：2026-03-12                                    ║
╠═══════════════════════════════════════════════════════╣
║ ✨ 新增主题                                            ║
║   • [新增] ClawBot CLI 命令                            ║
║   • [新增] Webhook 配置指南                            ║
║                                                       ║
║ 📝 更新内容                                            ║
║   • [更新] Gateway 架构说明                            ║
║   • [更新] Agent 协作模式                              ║
║                                                       ║
║ ⚠️  可能过时                                          ║
║   • [确认] 旧版配置方法（已在 v2.0废弃）               ║
╠═══════════════════════════════════════════════════════╣
║ 是否采纳变更建议？                                     ║
║   1. ✅ 全部应用                                       ║
║   2. 🔧 选择性应用                                     ║
║   3. ⏭️  跳过，稍后处理                                ║
╚═══════════════════════════════════════════════════════╝
```

### 步骤 4：用户选择处理

| 选择 | 操作 |
|------|------|
| 全部应用 | 自动创建新模块，更新现有内容，删除过时内容 |
| 选择性应用 | 逐项询问用户是否应用 |
| 跳过 | 仅更新缓存，不修改课程 |

### 步骤 5：执行变更并记录

如果用户确认应用变更：

1. **新增主题**：自动创建新模块文件
2. **更新内容**：更新对应的课程文件
3. **删除过时**：标记或删除过时内容
4. **记录变更**：添加到 `course-updates.md`

## course-updates.md 格式

```markdown
# 课程更新日志

## 2026-03-12 - 缓存刷新 v1.1 → v1.2

### 新增内容
- ✅ 已添加：ClawBot CLI 命令模块
  - 创建位置：`03-实战应用/clawbot-cli/`
  - 来源：缓存新增主题

### 更新内容
- ⏭️  已跳过：Gateway 架构说明
  - 用户选择暂不更新

### 删除内容
- 🗑️  已删除：旧版配置方法
  - 原因：已在 v2.0 废弃

### 待处理
- ⚠️  需要确认：新版本的配置迁移指南
```

## 变更类型处理

### 新增主题（NEW）

**操作**：建议添加新模块

```
检测到新增主题：ClawBot CLI 命令

建议：
• 创建新模块：03-实战应用/clawbot-cli/
• 生成模块 README.md、checklist.md、notes.md
• 添加到课程结构

是否创建？(y/n)
```

### 更新内容（UPDATED）

**操作**：提示更新现有模块

```
检测到内容更新：Gateway 架构说明

当前版本：v1.1
新版本：v1.2
变更：新增负载均衡章节

是否更新模块？(y/n)
```

### 过时内容（DEPRECATED）

**操作**：让用户决定是否保留

```
检测到可能过时的内容：旧版配置方法

原因：已在 v2.0 废弃
建议：删除或标记为"已废弃"

选项：
  1. 删除此内容
  2. 标记为"已废弃"并保留
  3. 暂不处理

你的选择：[等待用户输入]
```

# 🔄 同步工具

## 检查更新

**触发方式**（任意一种）：
- "检查更新" / "查看是否有更新" / "有更新吗"
- "模板有更新吗" / "检查新版本" / "版本更新"

**执行步骤**：

1. 检查 upstream 远程仓库是否配置
   - 未配置：显示配置指引并退出

2. 从 CHANGELOG.md 提取当前版本和上游版本

3. 显示版本比较结果：
   - 如有更新：显示版本号差异和同步指引
   - 如已是最新：显示相应提示

4. 显示同步指引：
   ```
   📋 同步方式（推荐使用 GitHub 原生同步）

   方法 1：GitHub UI（推荐）
     1. 打开你的 Fork 仓库页面
     2. 点击 "Sync fork" → "Update branch"
     3. 完成！

   方法 2：命令行
     git fetch upstream
     git merge upstream/main
   ```

## 课程模板更新机制

课程模板存储在 `.templates/modules/` 目录，会随上游同步更新。

用户初始化时，模板会复制到各模块目录：
- `.templates/modules/01-基础入门/ai-tools-fundamentals/README.md`
  → `01-基础入门/ai-tools-fundamentals/README.md`

由于模块目录的 README.md 被 gitignore 保护，用户可以：
- 自由修改自己的课程内容
- 如需获取最新课程，手动从 `.templates/modules/` 复制

# 📋 行为规范

## 书签探索行为规范

> **当书签状态为 🔵 探索中 时，AI 应遵循以下行为规范：**

### 探索阶段

1. **每次查询/操作完成后**：
   - 显示探索结果摘要（简洁，不超过 5 个要点）
   - ⭐ **关键：主动询问** "还有其他疑问吗？"
   - 等待用户响应，不要自动继续或返回主线

2. **用户回应的处理**：
   - 用户提出新疑问 → 继续探索，记录到探索笔记
   - 用户说"没有了"/"没问题" → 提示："可以说'完成书签'返回主线"
   - 用户说"完成书签" → 调用完成书签流程
   - 用户沉默/不明确 → 保持探索中状态，等待用户主动

3. **输出格式示例**：
   ```
   📚 探索结果：Tool Use 机制

   核心要点：
   • Tool Use 通过 function calling 实现
   • AI 主动调用外部工具，而非被动响应
   • 支持多工具并行调用

   ❓ 还有其他疑问吗？
      - 有疑问 → 直接告诉我
      - 没有了 → 说"完成书签"返回主线
   ```

### 禁止行为

- ❌ 不要在查询完成后自动说"让我们返回主线学习"
- ❌ 不要推测用户没有疑问
- ❌ 不要在没有用户确认的情况下结束探索

### 探索笔记记录（可选）

1. **触发条件**：用户明确说以下任一短语时触发
   - "记录一下"
   - "记到笔记里"
   - "保存这个"
   - "写到书签笔记"

2. **记录内容**：
   - 当前探索结果的要点摘要（与显示给用户的内容一致）
   - 记录时间戳

3. **记录位置**：
   - 添加到 `LEARNING_BOOKMARKS.md` 对应书签的"探索笔记"区域

4. **记录格式示例**：
   ```markdown
   ## 探索笔记

   ### 2026-03-07

   **Tool Use 机制探索**：

   - Tool Use 通过 function calling 实现
   - AI 主动调用外部工具，而非被动响应
   - 支持多工具并行调用
   ```

5. **记录后提示**：
   ```
   ✅ 已记录到书签笔记
   ```

6. **非自动记录**：
   - 用户未明确要求时，不自动记录
   - 避免文件频繁修改

## 理论知识展示行为规范

> 在学习过程中展示官方文档、概念解释、使用指南等理论知识时，AI 应遵循分块展示和用户确认机制，确保用户有充分时间理解内容。

### 适用范围

| 场景 | 示例 |
|------|------|
| 官方文档 | GitHub README、API 文档、技术规范 |
| 概念解释 | 术语定义、原理说明、架构描述 |
| 使用指南 | 安装步骤、配置说明、操作流程 |
| 理论知识 | 学习资料、教程内容、最佳实践 |

### 分块展示策略

**触发条件**：
- 展示内容超过 15 行
- 内容包含多个独立的逻辑单元（如多个概念、多个步骤）

**分块原则**：
- 按逻辑单元分块（如按概念、按步骤、按小节）
- 每块不超过 15 行（不含确认提示）
- 优先在自然段落边界分块

**示例**：
```
第 1/3 块：核心概念
[展示前 15 行内容]

═══════════════════════════════════
❓ 是否继续下一部分？
   - 回复任意内容继续
   - "跳过"进入下一步
   - "暂停"稍后继续
═══════════════════════════════════

[用户确认后...]

第 2/3 块：实现原理
[展示接下来的内容]
```

### 确认提示格式

```
═══════════════════════════════════
❓ 是否继续下一部分？
   - 回复任意内容继续
   - "跳过"进入下一步
   - "暂停"稍后继续
═══════════════════════════════════
```

### 用户响应处理

| 用户输入 | AI 行为 |
|---------|---------|
| 任意非指令内容（如"继续"、"好的"、"y"） | 继续展示下一块 |
| "跳过" | 跳过剩余理论内容，直接进入下一步操作 |
| "暂停" | 停止展示，告知用户可回复"继续"恢复 |
| "完成" | 结束理论展示，进入下一步操作 |

### 来源标注规范

| 内容类型 | 标注方式 | 示例 |
|----------|----------|------|
| 官方文档 | 文档名称 + 链接 | `来源：Claude Code 官方文档 https://code.claude.com/docs` |
| GitHub README | 仓库名 + 文件路径 | `来源：anthropics/claude-code/blob/main/README.md` |
| API 文档 | 文档标题 + URL | `来源：MCP SDK API Reference` |
| 缓存知识 | 缓存日期 | `来源：知识缓存 (2026-03-07)` |
| AI 生成内容 | 无需标注 | （直接展示） |

**标注格式**：
```
📚 来源：Claude Code 官方文档
https://code.claude.com/docs/getting-started

───────

[内容开始]
```

### 禁止行为

- ❌ 不要在展示完理论内容后自动继续下一步
- ❌ 不要推测用户已经读完内容
- ❌ 不要在用户未确认的情况下进入下一步
- ❌ 不要一次性展示超过 15 行的理论内容
- ❌ 不要因为 `accept edits on` 模式开启而省略理论确认

### 独立于 `accept edits on` 设置

> 重要说明：理论知识确认机制独立于文件编辑设置

- `accept edits on`：仅影响文件编辑操作，不影响理论展示
- 即使 `accept edits on` 开启，展示理论内容时仍需等待用户确认

# 📝 注意事项

## 通用注意事项

1. **用户确认**：修改多个文件前，总是先显示预览并等待用户确认
2. **日期格式**：统一使用 YYYY-MM-DD 格式
3. **文件路径**：所有路径相对于仓库根目录
4. **错误处理**：如果文件不存在或模块未找到，给出明确的错误提示

## 书签相关

- 书签探索期间，AI 应专注于书签疑问，避免讨论主线内容
- 探索结果使用要点列表，保持简洁（避免占用过多上下文）
- 每次查询后必须主动询问"还有其他疑问吗？"

## 缓存相关

- 缓存策略：每次询问用户，而非自动判断过期，给予用户完全控制
- 内容获取优化：拉取网页内容时使用 `retain_images=false` 跳过媒体文件，提升加载速度

## 同步相关

- 使用 GitHub 原生同步方式（"Sync fork" 按钮或命令行）
- 个人数据文件被 `.gitignore` 保护

## 状态标识

- 🟡 待探索
- 🔵 探索中
- 🟢 已完成
